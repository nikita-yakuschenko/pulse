# HTTP API модулей УПП (1С:Управление производственным предприятием)

Общее описание работы всех HTTP-модулей, документация API и инструкция по настройке и интеграции с внешними системами.

**Связанные документы:**
- **[HTTP_МОДУЛЬ_УПП_ПАМЯТКА.md](HTTP_МОДУЛЬ_УПП_ПАМЯТКА.md)** — внутренняя памятка для разработчиков модулей (архитектура кода, разбор параметров, стиль).
- В каждой папке **`<Модуль>/README.md`** — описание параметров и ответа конкретного эндпоинта.

---

## 1. Назначение

В каталоге **`bsl/`** собраны модули HTTP-сервисов для конфигурации 1С:УПП. Каждый модуль — один HTTP-метод (эндпоинт), который по запросу возвращает JSON с данными из базы (документы, справочники, отчёты). Документация в этом файле позволяет:

- подключаться к API из внешних приложений (веб, мобильные приложения, интеграции);
- однозначно настроить HTTP-сервис в конфигураторе 1С;
- понимать общие правила запросов и ответов без чтения кода.

---

## 2. Архитектура

- **Платформа:** HTTP-сервисы 1С (публикация в веб-сервере или встроенном HTTP-сервисе).
- **URL запроса** имеет вид: **`<базовый_URL>/<имя_HTTP_сервиса>/<метод>`**.  
  - **Базовый URL** — общий префикс до каталога HTTP-сервисов (например, `https://<хост>/<путь>/hs`).  
  - **Имя HTTP-сервиса** — сегмент пути, соответствующий одному модулю (например, `warehouses` для Склады).  
  - **Метод** — действие внутри сервиса (например, `get` для получения списка).
- В конфигураторе 1С для каждого такого пути настраивается **шаблон URL**, который вызывается при обращении по этому адресу. Обработчик (общий модуль) вызывает **экспортную функцию**, имя которой задаётся в настройках шаблона (часто совпадает с методом или с именем функции в коде).
- **Код модуля:** `bsl/<ИмяМодуля>/<ИмяМодуля>.bsl`. Внутри — одна функция с сигнатурой `Функция ИмяФункции(Запрос) Экспорт`.
- **Параметры запроса** берутся из тела запроса (**ПараметрыЗапроса**) и из query-string (**ПараметрыURL**). Сначала проверяется тело, при пустом значении — URL (единообразно во всех модулях).

---

## 3. Базовый URL и эндпоинты

**Структура URL запроса:**

```text
<базовый_URL>/<имя_HTTP_сервиса>/<метод>?параметр1=значение1&параметр2=значение2
```

- **Базовый URL** — префикс до каталога HTTP-сервисов (зависит от настроек публикации в вашей конфигурации). Например: `https://<хост>/<путь>/hs`. Вся часть после базового URL задаётся в конфигураторе 1С.
- **Имя HTTP-сервиса** — сегмент пути, обращение к конкретному сервису (одному модулю из каталога `bsl/`). В конфигураторе этому сегменту соответствует свой шаблон или подсервис.
- **Метод** — сегмент после имени сервиса (например, `get` — получение списка/данных). В конфигураторе по этому пути вызывается обработчик, который выполняет соответствующую функцию из кода модуля.

**Пример корректного запроса** (базовый URL и имена сервисов условные):

```text
GET <базовый_URL>/warehouses/get
GET <базовый_URL>/warehouses/get?code=01&name=Основной
```

Здесь `warehouses` — обращение к HTTP-сервису «Склады», `get` — метод (получить список); параметры передаются в query-string.

## 4. Общие соглашения API

### 4.1. Метод и передача параметров

- Допускаются запросы **GET** (параметры в query-string) и **POST** (параметры в теле, например form-urlencoded или JSON — в зависимости от того, как 1С отдаёт их в `ПараметрыЗапроса`).
- Параметры можно передавать в **query-string** и/или в **теле**. Сначала используется значение из тела запроса, при пустом — из URL (единая логика во всех модулях).

### 4.2. Форматы значений

- **Даты** (параметры `from`, `to`, `date`): поддерживаются форматы **dd.MM.yyyy**, **dd-MM-yyyy**, **yyyy-MM-dd**. В ответах даты всегда строка в формате **dd.MM.yyyy**.
- **Булево** (например `full`, `approved`): истина — `1`, `true`, `yes`, `да` (без учёта регистра).
- **Строковые фильтры** (code, name, responsible и т.п.): подстрока для поиска (в запросе к БД — ПОДОБНО, т.е. `%значение%`). Регистр при сравнении может не учитываться в части модулей (см. README модуля).
- **Параметр full:** в 1С ключи в URL могут приходить с разным регистром (`full`, `Full`, `FULL`). API учитывает все варианты.

### 4.3. Структура успешного ответа (HTTP 200)

Все эндпоинты возвращают JSON одного вида:

```json
{
  "data": [ ... ],
  "errors": [ ... ]
}
```

- **data** — массив записей (документы, справочники, остатки и т.д.). Структура элементов зависит от эндпоинта (см. README модуля).
- **errors** — массив строк с сообщениями об ошибках по отдельным операциям (например, по конкретному документу). Наличие элементов в `errors` не меняет код ответа: при успешном выполнении запроса всегда 200, а ошибки носят информационный характер.

### 4.4. Заголовки ответа

- **Content-Type:** `application/json; charset=utf-8`
- **Access-Control-Allow-Origin:** `*` (для вызовов из браузера и кросс-доменной интеграции)

### 4.5. Ошибка выполнения (HTTP 500)

При необработанном исключении в коде модуля возвращается:

- **Код:** 500  
- **Тело:** JSON вида `{"error": "текст ошибки из ОписаниеОшибки()"}`  
- Те же заголовки Content-Type и Access-Control-Allow-Origin.

Клиенту следует проверять код ответа и при 500 разбирать поле `error`.

---

## 5. Расширенный ответ (full)

Часть эндпоинтов (документы с табличными частями, связями, вложениями) поддерживают параметр **full**:

- **Без full** (или `full=false`): возвращается базовый набор полей документа + сводки по связям и вложениям (например, `Есть`, `Количество`), без выгрузки тяжёлых данных.
- **full=true**: к базовым полям добавляются полные табличные части, списки связанных документов, вложения (например, в Base64).

Подробности по каждому методу — в соответствующем README модуля.

## 6. Настройка и интеграция в 1С

### 6.1. Подключение модулей к конфигурации

1. В **конфигураторе** 1С:УПП откройте **HTTP-сервисы** (создание и настройка HTTP-сервисов выполняется только через конфигуратор).
2. Задайте **базовый URL** публикации (префикс до сегментов сервисов и методов).
3. Для каждого эндпоинта настройте путь вида **`<имя_HTTP_сервиса>/<метод>`** (например, `warehouses/get` для Склады). Имя сервиса и метод — те сегменты URL, по которым 1С будет вызывать обработчик.
4. В настройках шаблона (или подсервиса) укажите **обработчик** — общий модуль и **имя вызываемой функции** (например, `GetWarehouses` для `warehouses/get`). Код функции возьмите из `bsl/<ИмяМодуля>/<ИмяМодуля>.bsl` и поместите в этот общий модуль так, чтобы экспортная функция с нужным именем вызывалась при обращении по соответствующему URL.

Важно: имя вызываемой функции в коде (GetSOrders, GetWarehouses и т.д.) задаётся в настройках шаблона; путь в URL (`warehouses/get`, `supplier-orders/get` и т.п.) задаётся при конфигурировании HTTP-сервиса и может быть любым, удобным для вашей интеграции.

### 6.2. Проверка после настройки

1. Опубликуйте HTTP-сервис и перезапустите веб-сервер при необходимости.
2. Выполните запрос по URL в формате **`<базовый_URL>/<сервис>/<метод>`** (например, `<базовый_URL>/warehouses/get`) и убедитесь, что ответ 200 и в теле есть JSON с полями `data` и `errors`.
3. При необходимости передайте параметры в query-string (даты, код, full) и сверьте результат с описанием в README модуля.

### 6.3. Интеграция из внешнего приложения

- **Базовый URL** возьмите из настроек публикации HTTP-сервиса.
- **Полный URL запроса** — `<базовый_URL>/<имя_HTTP_сервиса>/<метод>` (см. таблицу в п. 3). Пример: `<базовый_URL>/warehouses/get`, `<базовый_URL>/sorders/get?from=01.01.2025&to=31.01.2025`.
- Параметры передавайте в query-string для GET или в теле для POST (с учётом того, как ваша 1С отдаёт их в `ПараметрыЗапроса`).
- Обрабатывайте ответ: при 200 разбирайте `data` и при необходимости `errors`; при 500 — поле `error` в JSON.
- Учитывайте CORS: заголовок `Access-Control-Allow-Origin: *` уже выставляется модулями, поэтому запросы из браузера на другой домен допустимы при отсутствии дополнительных ограничений на стороне 1С/веб-сервера.
